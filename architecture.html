OVERVIEW

What is Golfer's Edge?

A multi-city B2B2C SaaS platform for indoor golf facilities — handling bookings, loyalty, shop orders, invoicing, and GST compliance in one system.

🏈 For Members

  • Bay bookings with real-time slot availability
  • Hours packages — Birdie & Eagle plans
  • Loyalty points & tier progression (Bronze → Platinum)
  • Shop orders for beverages & merchandise
  • Events: tournaments, leagues, coaching sessions
  • Community feed, leaderboard & handicap tracking
  • Reward redemptions from loyalty catalogue
  • Profile management & booking history

⚙️ For Admins

  • Walk-in booking creation & management
  • GST-compliant invoice generation (GSTR-1 export)
  • Expense tracking & categorisation
  • Revenue reports, P&L, product profitability
  • Member management & role assignment
  • Loyalty config: rules, multipliers, milestones
  • Bay pricing, hour packages, payment gateways
  • Admin audit logs & background job monitoring

🏗 Platform Core

  • Multi-city scoping via AdminCityContext
  • Real-time calendar slot sync (Google Calendar)
  • bcrypt password auth + Supabase JWT
  • Row-Level Security on all 45 tables
  • HMAC-SHA256 Razorpay webhook verification
  • Deno Edge Functions with Zod validation
  • Atomic stored procedures (no partial state)
  • PWA manifest + service worker ready

FRONTEND STATE

State Management

Server State — React Query

  • All DB reads via useQuery + Supabase client
  • All writes via useMutation with queryClient.invalidateQueries
  • Key query domains: bookings, bays, orders, products, rewards, invoices, users, loyalty
  • Stale-while-revalidate pattern — no manual loading states
  • Calendar slots refetch every 30 seconds for live availability

Client / UI State

  • AuthContext: user session, profile completion flows (Apple relay, phone)
  • AdminCityContext: city selection (sessionStorage), role-scoped city list
  • No Redux / Zustand — context + React Query is sufficient at current scale
  • Theme: next-themes with dark mode default
  • Toast notifications via sonner
33+ Custom React Query Hooks — by domain
Bookings (8) Orders (4) Loyalty / Points (8) Admin / Finance (6) Config (5) Products (3) Community (2) Notifications (2)
All hook filenames follow the pattern use[Domain].ts — e.g. useBookings.ts, useLoyaltyConfig.ts, useAdminRevenue.ts. Each hook owns its own query key and cache invalidation strategy.

DATABASE FUNCTIONS

Atomic Stored Procedures

grant_site_admin_with_cities(user_id, cities[])

  • Atomically inserts user_roles row AND site_admin_cities rows in one transaction
  • Prevents partial state: role granted but city assignment fails
  • SECURITY DEFINER — service role only, not callable from browser
  • Called by manage-roles edge function

cancel_booking_with_clawback(booking_id, cancelled_by)

  • Row-locks booking, marks as cancelled
  • Sums allocation points for that booking_id
  • Inserts negative points_transactions (redemption)
  • Decrements profile.points (floor: 0 — never negative)
  • All in one transaction — impossible to have cancelled booking with unclaimed points

audit_log_trigger()

  • AFTER trigger on INSERT / UPDATE / DELETE of critical tables
  • Records: changed_by (auth.uid()), table_name, operation, old_values, new_values as JSONB
  • Append-only — no UPDATE/DELETE policy on audit_log
  • Active on: user_roles, payment_gateways, bay_pricing, admin_config, bookings (status only), invoices

RLS Helper Functions

  • has_role(uid, role) — checks user_roles table
  • is_admin_or_site_admin(uid) — used in all write RLS policies
  • has_city_access(uid, city) — scopes site_admin to their cities
  • increment_user_points(user_id, delta)
  • decrement_user_points_safe(user_id, delta) — floors at 0

SECURITY · AUTH

Authentication & Authorisation

Authentication

  • Supabase JWT — every request carries Bearer token in Authorization header
  • Apple OAuth: relay email detection triggers profile completion modal
  • Phone completion modal for missing phone numbers
  • localStorage session with auto-refresh (supabase client config)
  • bcrypt password for admin setup and password change flows
  • create-admin-user reads password from ADMIN_AUTH_PASSWORD env var

Role Hierarchy

  • admin — Full access: all cities, all tabs, all reports, role manager, settings
  • site_admin — City-scoped: assigned cities only, limited reports (SiteAdminPermissionsCard)
  • user — Member access: dashboard, bookings, shop, rewards, profile
  • Roles stored in user_roles table, scoped by city for site_admin
  • Role checks embedded in RLS policies — no frontend-only role guards
RLS Helper Functions (SECURITY DEFINER)
has_role(uid, role) is_admin_or_site_admin(uid) has_city_access(uid, city)
All three RLS helpers are SECURITY DEFINER — they run with the privileges of the function owner, not the calling user. This allows them to query user_roles and site_admin_cities without exposing those tables via RLS bypass.

SECURITY · RLS · APRIL 2026

Row-Level Security Hardening

Before (Critical Bug)

-- ANY logged-in user could insert! CREATE POLICY "Authenticated users can insert events" ON public.events FOR INSERT TO authenticated WITH CHECK (true); ← no role check
  • Same WITH CHECK (true) on: products, rewards, earn_methods (INSERT/UPDATE/DELETE)
  • community_posts UPDATE: USING(true) — any user could edit any post
  • No DELETE policies on community_posts — all authenticated could delete

After (Fixed — Migration 20260409100001)

-- Only admins can write CREATE POLICY "Admins can insert events" ON public.events FOR INSERT TO authenticated WITH CHECK (is_admin_or_site_admin(auth.uid()));
-- Own posts only ON community_posts FOR UPDATE TO authenticated USING (auth.uid() = user_id) WITH CHECK (auth.uid() = user_id); -- + admin override policy
Migrations 20260409100001 and 20260409100002. All business-data write operations now require is_admin_or_site_admin() — regular members cannot corrupt platform data.

SECURITY · AUTH HARDENING

Password & Rate Limiting

Password Security

  • Old: plain-text stored in admin_config table, direct === comparison (timing attack vulnerable)
  • New: bcrypt hash via deno.land/x/bcrypt@v0.4.1, constant-time safeCompare()
  • Legacy migration: plain-text auto-upgraded to bcrypt on next successful login
  • Strong policy: 12+ chars, uppercase, lowercase, number, special char
  • create-admin-user reads from ADMIN_AUTH_PASSWORD env var (not DB hash)
  • Password change: rate-limited + bcrypt rehash on every change

Rate Limiting

  • Table: rate_limit_attempts (identifier, action, attempted_at)
  • Max: 5 attempts per 15 minutes per identifier
  • Returns HTTP 429 when limit exceeded
  • Index: (identifier, action, attempted_at) for fast window queries
  • Survives function restarts — stored in DB, not in-memory
  • setup-admin: IP-based rate limit (x-forwarded-for header)
  • change-admin-password: user_id-based rate limit
Migrations 20260409100003 and 20260409100004. The bcrypt upgrade is backward-compatible: if the stored hash does not start with $2, it is treated as plain-text, verified, then immediately re-hashed and stored.

SECURITY · PAYMENTS

Payment Security

Webhook Verification (razorpay-webhook)

  • 1. Read raw body before JSON parse — preserves byte-exact payload for signature
  • 2. Load per-city webhook_secret from payment_gateways or RAZORPAY_WEBHOOK_SECRET env
  • 3. HMAC-SHA256(secret, rawBody) using Web Crypto API (crypto.subtle)
  • 4. Constant-time hex comparison against x-razorpay-signature header
  • 5. Reject with HTTP 403 on signature mismatch
  • 6. Idempotency: INSERT with UNIQUE razorpay_event_id — skip if already processed

Secret Management

  • api_secret column: REVOKE SELECT from authenticated role (service-role only)
  • Edge function reads: RAZORPAY_SECRET_<CITY_SLUG> env var (e.g. RAZORPAY_SECRET_BANGALORE)
  • Fallback to DB column during migration period
  • payment_gateways.city_slug: auto-generated computed column
  • payment_status tracked on both bookings and orders rows
  • Migration 20260409100008: DROP COLUMN api_secret (uncomment after migration)
Migrations 20260409100005–20260409100008 cover: payment_events idempotency table, HMAC webhook logic, api_secret column revocation, and payment_status columns on bookings/orders.

LOYALTY ENGINE

Loyalty System

Earning

  • Event types: walkin, birdie_usage, eagle_usage, coaching, practice, renewal
  • Base rate × stacking multipliers from multipliers table
  • Multiplier conditions: off-peak (auto-detected from bay peak_start/end), coaching status, first-visit bonus, tier-based (eagle/birdie), renewal
  • Points credited via calculate-loyalty-points edge function
  • booking_id FK on points_transactions enables atomic clawback on cancellation
  • Admin can manually gift rewards to any user

Tiers & Redemption

  • Bronze (0–1,000) → Silver (1,000–5,000) → Gold (5,000–10,000) → Platinum (10,000–25,000)
  • Milestones: bonus points at hours-logged thresholds (user_progress table)
  • Rewards catalogue: point cost, daily cap, usage-gate (unlock after X% plan hours used)
  • Auto-gifts: trigger_event rules (e.g. on signup) with max_per_user cap
  • process-auto-gifts edge function processes rules on each trigger event
  • gifted_rewards table tracks which rewards were auto-gifted to whom

CONVENTIONS

Developer Conventions

Frontend Patterns

  • Query hooks: useQuery + .from('table').select('*')
  • Mutations: useMutation + onSuccess invalidateQueries
  • Type cast for new tables: supabase.from('new_table' as any) — NOT (supabase as any).from()
  • Error handling: throw error from queryFn; React Query surfaces to UI
  • Zod forms: react-hook-form + zodResolver for all admin forms
  • Toast: toast.success() / toast.error() via sonner

Edge Function Patterns

  • Zod schema parse at function entry; reject HTTP 400 on failure
  • Body size check: contentLength > MAX_BODY_BYTES → HTTP 413
  • CORS: Deno.env.get('ALLOWED_ORIGIN') || '*'
  • Error logging: logJobFailure(adminClient, jobType, message, context)
  • Auth: userClient.auth.getUser() → verify role via adminClient.rpc('has_role')
  • Rate limit check before any processing — fail fast pattern
Bcrypt in Deno: deno.land/x/bcrypt@v0.4.1. Zod in Deno: esm.sh/zod@3. Rate limiting uses the rate_limit_attempts DB table — not in-memory — so it survives function restarts and cold starts across all Deno isolates.
